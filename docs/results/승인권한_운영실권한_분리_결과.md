# 승인권한 / 운영실권한 분리 결과

## 1. 목적

- 승인권한과 운영실 수정권한이 서로 섞이지 않게 분리
- `체크된 권한만 가져간다`는 원칙에 맞게 정책을 정리
- 마스터는 예외적으로 기존 전권 유지

쉽게 말하면:
- `승인 도장`과 `운영실 수정 펜`을 따로 보관하고,
- 체크된 사람에게만 그 도구를 주는 구조로 바꾼 것이다.

## 2. 기존 문제

기존에는 승인권한이 있는 사용자가 일부 상황에서 관리자처럼 해석될 여지가 있었다.

즉:
- 승인권자에게는 원래 `승인/반려`만 열려야 하는데
- 코드상 `타인 대상 수정/생성/삭제` 권한까지 이어질 수 있는 연결이 있었다.

## 3. 조치 내용

### 3.1 서버 권한 정책 정리

변경 전:
- `approveScope`가 있으면 `canManageTargetUserForActor`도 일부 true가 될 수 있었음

변경 후:
- `canManageTargetUserForActor`는 이제 아래 권한만 본다
  - 마스터
  - 권한/설정 접근
  - 구성원 관리
  - 운영실 접근
- 승인권한(`approveScope`)만으로는 타인 관리 권한이 열리지 않음

반영 파일:

## 4. 의미

### 4.1 승인권한만 있는 사람

- 승인/반려 가능
- 운영실 대리 등록/수정/삭제 불가

### 4.2 운영실 권한만 있는 사람

- 운영실 접근 가능
- 대리 등록/수정/삭제 가능
- 승인/반려는 별도 승인권한 없으면 불가

### 4.3 둘 다 가진 사람

- 승인도 가능
- 운영실 작업도 가능

### 4.4 마스터

- 기존처럼 모든 권한 유지

## 5. 정책 검증

로컬 정책 검증 기준:

- 승인권한만 있는 사용자
  - `canManageTargetUserForActor = false`
  - `admin_create/admin_edit = 불가`

- 운영실 권한이 있는 사용자
  - `canManageTargetUserForActor = true`
  - `admin_create/admin_edit = 가능`

즉:
- 승인권한과 운영실 수정권한이 서버 정책상 분리되었다.
