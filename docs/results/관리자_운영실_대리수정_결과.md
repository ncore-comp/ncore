# 관리자 운영실 대리수정 결과

업데이트: 2026-03-25

## 1. 구현 목적

운영실에서 관리자가 기존 연차/반차/시간차/특별휴가 기록을 `삭제 후 재등록`하지 않고,
기존 요청 문서를 직접 정정할 수 있게 만드는 것이 목적이었다.

핵심 원칙:

- 결재 아님
- 승인/반려 절차 없음
- 메일 발송 없음
- 기존 DB 기록 직접 수정
- status는 그대로 유지

## 3. 가장 중요한 수정 포인트

### A. editingId 초기화 버그 수정

문제:

- 운영실 `수정` 버튼을 누르면 `editingId`를 세팅했지만
- 모달 초기화(`initModal`)에서 다시 `null`로 지워지고 있었다

결과:

- 수정 모드처럼 보여도 실제 저장은 신규 등록처럼 동작할 수 있었다

조치:

- `initModal(options)`가 `editId`를 받아 유지하도록 수정
- `editAdminManagedRequest()`는 `editMode: true, editId: ...`로 모달을 연다

### B. 김효창 사례 반영

문제:

- 관리자 대리 수정 모달의 `시간차(퇴근)` 퇴근시간 표시가
- 현재 로그인한 관리자 기준으로 잡힐 수 있었다

원하는 규칙:

- `대상 직원 + 해당 날짜의 현재 분기 근무시간` 기준

조치:

- `syncRequestTypeDefaults()`에서
  - `app.currentUser`
  - 가 아니라
  - `app.getRequestModalTargetUser()`
  기준으로 `퇴근 시간`을 계산하도록 수정

결과:

- 김효창의 현재 퇴근시간이 16시로 바뀐 상태라면
- 관리자 대리 수정 모달도 `16:00`으로 표시된다
- 저장 결과 `timeRange`도 그 기준으로 다시 계산된다

## 4. 실제 검증 결과

검증 스크립트:

- [scripts/verify-admin-proxy-edit.js](/D:/MYDOC/Desktop/vscode/Web/firebase/ncore_web/ncore-main/scripts/verify-admin-proxy-edit.js)

실행 방식:

- 운영 사이트 [https://ncore.web.app](https://ncore.web.app) 로그인
- 운영실 진입
- 김효창 대상 임시 `시간차(퇴근)` 승인건 생성
- 운영실 카드 `수정`
- 사용시간 `2시간 -> 3시간`, 사유 `Refresh -> 여행` 수정
- 저장 후 새로고침/재로드 확인
- 테스트 데이터 삭제

실제 통과 결과:

- 모달 제목: `관리자 대리 수정`
- 대상 직원: `김효창 (파츠북팀)`
- 퇴근시간 표시: `16`
- 저장 후 status: `approved` 유지
- 저장 후 reason: `여행`
- 저장 후 timeRange: `13:00~16:00`
- 관리자 로그: `AdminRequestEdit` 확인

## 5. 최종 판단

운영실 대리수정 기능은 다음 조건을 충족한다.

1. 기존 기록을 직접 수정할 수 있다
2. 승인 절차를 다시 타지 않는다
3. status를 유지한다
4. 대상 직원의 현재 근무시간 기준으로 시간차(퇴근) 범위를 다시 계산한다
5. 관리자 정정 로그가 별도로 남는다

- 수정
