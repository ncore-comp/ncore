## 문제 요약
## Problem Summary
- 현재 구조는 `featureBoard` 전역 스위치와 사용자별 실제 게시판 접근 판정이 섞여 있습니다.
- The current structure mixes the global `featureBoard` switch with per-user board access decisions.

- 그래서 어떤 사용자는 상단에 `공지사항` 버튼은 보이지만, 실제 글 목록은 빈 배열을 받는 모순이 생깁니다.
- Because of that, some users can see the `공지사항` button in the top nav, but still receive an empty list of posts.

- 이 문제를 가장 안전하게 풀려면, 게시판도 다른 기능처럼 `개인 권한`으로 분리해야 합니다.
- The safest fix is to separate board access into explicit `per-user permissions`, just like the other feature groups.

쉽게 말하면:
전구 스위치 하나(`featureBoard`)로 회사 전체를 켜고 끄는 것과, 사람마다 열쇠(`읽기`, `쓰기`)를 주는 것을 분리해야 합니다.

In simple terms:
We need to separate the company-wide light switch (`featureBoard`) from each person's key (`read`, `write`).

## 권한 모델 설계
## Permission Model Design

### 1. 새 권한 필드
### 1. New Permission Fields
- `permissions.boardRead`
- `permissions.boardWrite`

설명:
- `boardRead`: 공지사항 보기 권한
- `boardWrite`: 공지사항 글쓰기 권한

Meaning:
- `boardRead`: permission to view notice-board posts
- `boardWrite`: permission to create notice-board posts

### 2. 기존 전역 스위치와의 관계
### 2. Relationship With the Existing Global Switch
- 기존 `featureBoard`는 유지합니다.
- Keep the existing `featureBoard`.

- 역할 분리:
- Responsibility split:
  - `featureBoard` = 회사 전체 on/off 스위치
  - `boardRead`, `boardWrite` = 개인 권한

판정 방식:
1. `featureBoard === false`
   - 전원 비노출
   - no one sees the board
2. `featureBoard === true`
   - 각 사용자의 `boardRead`, `boardWrite`로 세부 권한 판단
   - actual behavior follows each user's `boardRead` / `boardWrite`

### 3. 기본값 정책
### 3. Default Policy
- `master`
  - `boardRead = true`
  - `boardWrite = true`

- `ceo`, `team_leader`, `part_leader`, `employee`
  - 권장 기본값:
    - `boardRead = true`
    - `boardWrite = false`

이유:
- 지금 공지사항은 회사 전체 공지 성격이 강합니다.
- The notice board currently behaves like a company-wide announcement board.

- 현재 실질 이슈도 “직원이 읽지 못하는 것”이 먼저 드러났습니다.
- The immediate issue is that employees cannot read posts consistently.

### 2. 공지사항 페이지 진입
### 2. Entering the Notice Page

  - 읽기 권한 없으면 차단
  - 안내 문구 표시

### 3. 글쓰기 버튼
### 3. Write Button
- `boardWrite === true`일 때만 `글쓰기` 버튼 노출
- Show `글쓰기` only when `boardWrite === true`

### 4. 수정/삭제 권한
### 4. Edit/Delete Permission
- 추천 기준:
  - `master`는 항상 가능
  - 일반 사용자는 `boardWrite === true` 이면서 `본인 글`일 때만 가능

즉:
- 읽기 권한만 있는 직원은 글은 읽을 수 있지만, 새 글 작성/기존 글 수정은 못 합니다.

In short:
- Users with read-only access can read posts but cannot create or edit them.
