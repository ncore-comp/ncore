## 1. 이 문서의 목적

이번 테스트는 단순 체크리스트를 다시 보는 단계가 아닙니다.

쉽게 비유하면:
- 지금까지 테스트는 `부품 검사`
- 이번 테스트는 `하루 모의 영업`

즉,
- 직원이 실제로 로그인해서 신청하고
- 팀장이 결재하고
- 대표가 전체 상황판을 보고
- 마스터가 구성원을 관리하고
- 두 사람이 동시에 같은 작업을 눌렀을 때도 시스템이 안 흔들리는지
를 `실제 운영 흐름`으로 확인하는 계획입니다.

## 2. 목표

이번 실제 시뮬레이션의 목표는 아래 4개입니다.

1. 주요 기능이 실제 사용자 흐름 안에서 전부 동작하는지 확인
2. 역할별 권한이 운영 기준과 정확히 맞는지 확인
3. 동시접속 / 동시신청 / 동시등록 / 동시승인 상황에서 데이터가 안전한지 확인
4. Firestore 기록과 accessLogs까지 실제로 남는지 확인

## 3. 테스트 원칙

1. 테스트 기준은 **운영 사이트**입니다.
   - [https://ncore.web.app](https://ncore.web.app)
2. 가능한 한 `로그인 -> 작업 -> 반영 확인 -> 로그 확인` 순서로 봅니다.
3. 테스트용 데이터는 접두어를 붙입니다.
   - 예: `SIM_20260323_01`
4. 테스트 후 생성한 임시 데이터는 정리합니다.
5. 실패하면
   - 화면 실패인지
   - API 실패인지
   - Firestore 기록 실패인지
   - 동시성 충돌인지
   를 구분합니다.

## 4. 역할과 준비 계정

### 1. 일반직원
역할:
- 본인 신청
- 반차/시간차 신청
- 특별휴가 신청
- 게시판 작성/조회
- 비밀번호 변경

준비:
- 일반직원 A
- 일반직원 B

### 2. 팀장
역할:
- 팀원 요청 결재
- 반려 사유 입력
- 결재대기 확인

### 3. 대표
역할:
- 전체 상황판 확인
- 자리배치도 확인
- 전사 현황 확인

### 4. 마스터
역할:
- 사용자 등록/수정/삭제
- 특별휴가 부여
- 메일 라우트 저장
- 게시판/권한/설정 관리

## 5. 사전 준비

### 브라우저 준비
- 일반 브라우저 1개
- 시크릿 창 2개 이상
- 가능하면 다른 브라우저 1개 추가

비유:
- 계산대 하나만 보는 게 아니라
- 여러 창구를 동시에 여는 상황을 만들기 위함입니다

### 데이터 준비
- 테스트용 일반직원 2명
- 팀장 1명
- 대표 1명
- 마스터 1명
- 특별휴가 부여 가능한 상태
- 테스트용 게시글 제목 규칙
  - 예: `SIM_BOARD_20260323`

### 확인 도구
- Firestore Console
- accessLogs 확인 경로
- 필요 시 Playwright / API 스크립트 보조

## 6. 전체 테스트 구조

테스트는 6개 Phase로 나눕니다.

1. 오픈 전 점검
2. 일반직원 하루 업무
3. 팀장 결재 흐름
4. 대표/전체 상황판
5. 마스터 운영 업무
6. 동시성 집중 테스트

## 7. Phase 1. 오픈 전 점검

목적:
- 영업 시작 전에 사이트가 정상인지 확인

시나리오:
1. 메인 페이지 첫 로드
2. 로그인 폼 노출 확인
3. 새로고침 후 로드 정상
4. 모바일 화면 진입
5. 콘솔 치명적 에러 없음

통과 기준:
- 첫 로드 체감 정상
- 로그인 화면 정상
- 새로고침 후 깨짐 없음
- 모바일에서 기본 진입 가능

증적:
- 화면 캡처
- 콘솔 에러 여부

## 8. Phase 2. 일반직원 하루 업무

목적:
- 일반직원이 실제 하루 동안 쓰는 기능이 끊김 없이 되는지 확인

### 시나리오 2-1. 일반 연차
1. 일반직원 A 로그인
2. 연차 신청 생성
3. 생성 후 달력/최근 신청 내역 확인
4. 연차 수정
5. 수정 반영 확인

통과 기준:
- Firestore 문서 생성/수정
- 화면 즉시 반영

### 시나리오 2-2. 반차 / 시간차
1. 반차(오전) 신청
2. 반차(오후) 신청
3. 시간차(퇴근) 신청
4. 시간차(외출) 신청

통과 기준:
- 각 타입별 저장 정상
- 시간/구간 값 정상

### 시나리오 2-3. 특별휴가 / 경조휴가
1. 관리자 부여 여부 확인
2. 일반직원 A 로그인
3. 특별휴가 노출 확인
4. 경조휴가 신청
5. 주말/공휴일 허용 규칙 확인

통과 기준:
- 부여된 항목만 노출
- 경조휴가 규칙 정상

### 시나리오 2-4. 게시판
1. 일반직원 A 게시글 작성
2. 작성 후 목록 반영 확인
3. 본인 작성글 수정

통과 기준:
- Firestore 반영
- 화면 목록 반영

### 시나리오 2-5. 로그아웃
1. 일반직원 A 로그아웃
2. 세션 종료 확인

통과 기준:
- 로그아웃 후 보호 화면 접근 차단

## 9. Phase 3. 팀장 결재 흐름

목적:
- 결재 담당자가 실제로 처리할 때 상태 전이가 안정적인지 확인

### 시나리오 3-1. 승인
1. 팀장 로그인
2. 결재대기 목록 확인
3. 일반직원 A 요청 승인
4. 직원 화면에서 승인 반영 확인

### 시나리오 3-2. 반려
1. 일반직원 B 요청 생성
2. 팀장 로그인
3. 요청 반려
4. 반려 사유 입력
5. 직원 달력 hover 툴팁에서 사유 표시 확인

### 시나리오 3-3. 취소 흐름
1. 승인된 요청에 대해 직원이 취소 요청
2. 팀장이 취소 승인

통과 기준:
- 승인 / 반려 / 취소 흐름 모두 정상
- 반려 사유 표시 정상
- 권한 없는 요청은 차단

## 10. Phase 4. 대표 / 전체 상황판

목적:
- 대표가 실제 운영 현황을 보는 흐름이 안정적인지 확인

시나리오:
1. 대표 로그인
2. 전체 상황판 진입
3. 필터 변경
4. 날짜 클릭
5. 자리배치도 열림
6. 우측 당일 부재 인원 리스트 확인
7. 닫기 버튼 확인

체크 포인트:
- SVG 정상 로드
- 날짜별 인원 일치
- 좌석 색상 일치
  - 출근 초록
  - 연차/특별연차 빨강
  - 반차/시간차 노랑

통과 기준:
- 대표가 별도 설명 없이도 읽을 수 있는 상태

## 11. Phase 5. 마스터 운영 업무

목적:
- 관리자가 실제 운영을 감당할 수 있는지 확인

### 시나리오 5-1. 구성원 등록
1. 마스터 로그인
2. 새 구성원 등록
3. 등록 확인창 확인
4. Firestore 반영 확인

### 시나리오 5-2. 구성원 수정
1. 방금 만든 구성원 수정
2. 재로딩 후 유지 확인

### 시나리오 5-3. 구성원 삭제
1. 삭제 버튼 클릭
2. 2회 확인창 확인
3. 실제 삭제 반영 확인

### 시나리오 5-4. 특별휴가 부여
1. 특정 직원에게 특별휴가 부여
2. 직원 로그인 후 신청 모달 노출 확인

### 시나리오 5-5. 메일 라우트
1. To/CC 설정 저장

### 시나리오 5-6. 게시판/설정
1. 게시판 공지글 작성
2. 권한/설정 저장
3. 저장 후 재로딩 유지

통과 기준:
- 인사/총무/운영 업무가 모두 정상

## 12. Phase 6. 동시성 집중 테스트

이 Phase가 가장 중요합니다.

쉽게 비유하면:
- 조용한 시간 테스트가 아니라
- 점심 피크 시간에 사람들이 몰린 상황을 흉내 내는 것입니다

### 6-1. 동시 로그인

참여:
- 일반직원 A
- 일반직원 B
- 팀장
- 대표
- 마스터

방식:
- 5개 창에서 동시에 로그인

통과 기준:
- 모두 로그인 성공
- 세션 충돌 없음

### 6-2. 동시 신청

시나리오 A:
- 일반직원 A와 B가 서로 다른 날짜 연차를 동시에 신청

기대:
- 모두 성공

시나리오 B:
- 일반직원 A가 같은 날짜를 2개 창에서 동시에 신청

기대:
- 1건만 성공
- 나머지는 중복 차단

### 6-3. 동시 승인/반려

시나리오:
- 같은 요청을 팀장과 대표/마스터 창에서 동시에 처리

기대:
- 한쪽만 성공
- 다른 쪽은 충돌 처리
- 최종 상태 1개로 수렴

### 6-4. 동시 구성원 등록

시나리오 A:
- 서로 다른 ID 2건 동시 등록

기대:
- 둘 다 성공

시나리오 B:
- 같은 ID 2건 동시 등록

기대:
- 1건만 반영되거나 충돌 처리

### 6-5. 동시 로그 기록

시나리오:
- 여러 계정이 동시에 로그인/저장/로그아웃

기대:
- accessLogs 누락 없음
