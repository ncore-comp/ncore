# 관리자 운영실 대리수정 계획

업데이트: 2026-03-25

## 1. 왜 필요한가

현재 운영실에서는 관리자가 다음 두 가지만 할 수 있다.

1. 다른 직원의 연차/반차/시간차/특별휴가를 대리 등록
2. 잘못 들어간 기록을 삭제

하지만 실제 운영에서는 `삭제 후 재등록`보다 `기존 기록을 바로 수정`해야 하는 경우가 있다.

예:

- 날짜는 맞는데 종류만 잘못 들어간 경우
- 사유는 맞는데 시간차 사용 시간이 잘못 들어간 경우
- 직원 근무시간이 나중에 수정되어 과거 기록을 운영상 정정해야 하는 경우

쉽게 비유하면:

- 지금은 운영실에 `지우개`만 있고 `수정 펜`이 없다
- 그래서 잘못된 장부를 고치려면 지우고 다시 쓰는 번거로운 방식이다
- 앞으로는 장부 위에서 바로 수정 펜으로 고치는 기능이 필요하다

## 2. 이번 기능의 성격

이 기능은 `결재`가 아니다.

- 승인/반려 절차를 타지 않는다
- 결재 메일을 보내지 않는다
- DB에 저장된 기존 요청 문서를 직접 정정한다
- 상태(status)는 그대로 유지한다

즉:

- `승인` = 결재 도장
- `관리자 대리수정` = 장부 정정

## 3. 핵심 원칙

### 3-1. 직접 수정 원칙

- 운영실의 `수정` 버튼은 `관리자 직접 정정 모드`로 동작한다
- 기존 요청 문서를 덮어쓴다
- `approved`는 수정 후에도 `approved`
- `pending`은 수정 후에도 `pending`
- `rejected`는 수정 후에도 `rejected`

### 3-2. 로그 기록 원칙

- 누가
- 누구의 요청을
- 어떤 값에서 어떤 값으로 바꿨는지
- 별도 운영 로그로 남긴다

추천 로그 타입:

- `admin_request_edit`

### 3-3. 승인 흐름과 완전 분리

- 승인/반려 버튼 로직과 섞지 않는다
- 운영실 수정은 별도 플래그로 처리한다
- 서버에서도 `결재 액션`과 `관리자 직접 수정 액션`을 분리한다

## 4. UX 방향

### 추천안

기존 연차 신청 모달을 재사용한다.

흐름:

1. 운영실 하단 기록 카드에서 `수정` 클릭
2. 기존 연차 신청 모달을 `관리자 대리 수정` 모드로 오픈
3. 기존 값이 미리 채워져 있음
4. 수정 후 `수정완료`
5. 기존 요청 문서를 직접 업데이트

비유:

- 새 종이를 만드는 게 아니라
- 기존 신청서를 수정 책상 위에 올리고 빨간 펜으로 정정하는 방식

### 수정 가능한 항목

- 대상 직원
- 종류
- 날짜
- 시간차 사용 시간
- 시간차 퇴근 시간 표시
- 사유

### 수정 불가능하거나 자동 결정되는 항목

- request id
- createdAt
- 원본 작성자 identity
- status 자체는 별도 수정 대상이 아님

## 5. 가장 중요한 규칙: 김효창 사례

김효창 사례를 기준으로 규칙을 고정한다.

상황:

- 김효창의 퇴근시간이 예전에 `17시`로 잘못 입력되어 있었음
- 현재는 `16시`로 수정된 상태
- 운영실에서 김효창의 과거 `시간차(퇴근)` 기록을 수정할 때
- 수정 모달은 `현재 김효창의 근무시간 기준`을 반영해야 함

즉:

- 관리자 본인의 근무시간 기준이 아님
- 과거 저장값을 그대로 보여주는 것도 아님
- `대상 직원 + 해당 날짜의 현재 분기 근무시간 규칙`을 기준으로 다시 계산해야 함

예:

- 대상 직원: 김효창
- 수정 대상 날짜: 2026-03-20
- 현재 분기 퇴근시간: 16:00
- 시간차(퇴근) 사용시간: 2시간

그러면 수정 모달과 저장 결과는 `14:00~16:00` 기준으로 맞춰져야 한다.

### 이 규칙의 의미

- 운영실 수정은 `과거 신청서 원본 보기`가 아니라
- `현재 운영 기준에 맞는 정정 입력`
이다

## 6. 데이터 처리 방식

### 6-1. 프런트

운영실 카드의 `수정` 버튼을 누르면:

- 대상 요청을 찾는다
- `관리자 대리 수정 모드` 플래그를 켠다
- 대상 직원 정보를 모달에 주입한다
- 기존 값(종류/날짜/사유/시간차 사용 시간)을 미리 채운다
- `시간차(퇴근)`인 경우
  - 대상 직원 기준으로 퇴근시간 표시를 다시 계산한다

### 6-2. 서버

서버는 이것을 `approve/reject`가 아니라 `admin_edit` 또는 별도 `admin_request_edit`로 처리한다.

권장 규칙:

- status 변경 없음
- content fields만 갱신
- updatedAt 갱신
- admin edit log 저장

### 6-3. timeRange 재계산

시간차(퇴근) 수정 시:

- 대상 직원
- 수정 대상 날짜
- 현재 분기 근무시간

기준으로 `timeRange`를 다시 만든다.

즉:

- 예전 저장값 `15:00~17:00`을 그대로 쓰지 않고
- 현재 운영 기준으로 새로 계산한다

## 10. 완료 기준

아래 조건을 모두 만족하면 완료로 본다.

1. 운영실 카드에 `수정` 버튼이 보인다
2. 수정 모달이 정상적으로 열린다
3. 종류/날짜/사유/시간차 사용시간 수정이 된다
4. 시간차(퇴근) 수정 시 대상 직원 현재 기준 퇴근시간이 반영된다
5. 승인 여부(status)는 유지된다
6. DB에 실제로 값이 반영된다
7. 새로고침 후에도 수정값이 유지된다
8. 관리자 수정 로그가 남는다

## 11. 결론

이번 기능은 `결재`가 아니라 `정정`이다.

즉 운영실 수정은

- 승인 도장 기능이 아니라
- 관리자 빨간 펜 기능

으로 분리해야 하며,
김효창 사례처럼 `대상 직원의 현재 분기 근무시간 기준`이 반영되어야 운영상 맞다.
