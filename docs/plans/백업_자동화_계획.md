## 1. 왜 필요한가

현재는 아래가 이미 가능합니다.
- 코드 복구: 로컬 이전 버전 파일, GitHub 이력
- 데이터 비교: [compare-firestore.js](/D:/MYDOC/Desktop/vscode/Web/firebase/ncore_web/ncore-main/scripts/compare-firestore.js)
- 데이터 복구: [import-firestore.js](/D:/MYDOC/Desktop/vscode/Web/firebase/ncore_web/ncore-main/scripts/import-firestore.js)
- 복구 리허설: 완료

하지만 이건 아직 `수동 소방훈련`에 가깝습니다.

쉽게 비유하면:
- 지금은 사람이 직접 장부를 복사해서 금고에 넣을 수 있는 상태
- 자동화는 `매일 밤 자동으로 장부를 복사하고, 실패하면 바로 알려주는 야간 직원`을 두는 것입니다

즉 백업 자동화의 목적은:
1. 사람 손을 덜 타게 만들기
2. 백업 누락 가능성을 줄이기
3. 복구 가능한 시점을 더 촘촘하게 확보하기

## 2. 자동화의 목표

최소 목표:
1. Firestore 현재 상태를 정기적으로 저장
2. 저장 성공/실패 여부를 기록
3. 최근 백업 파일 목록을 관리

권장 목표:
4. 컬렉션 수량 요약도 같이 남기기
5. compare 결과도 주기적으로 저장
6. 실패 시 관리자에게 알림

## 3. 백업 대상

### 필수
- `users`
- `requests`
- `boardPosts`
- `holidays`
- `specialLeaveTypes`
- `userSpecialLeaves`
- `mailRoutes`

### 선택
- `accessLogs`

설명:
- `accessLogs`는 계속 누적되므로 용량이 빨리 커질 수 있습니다
- 필요하면 `전체 백업`과 `핵심 운영 데이터 백업`을 나눌 수 있습니다

## 4. 추천 방식

추천은 2단계 구조입니다.

### 1단계. 로컬 JSON 백업 자동화

현재 이미 수동으로 만든 형태를 자동화합니다.

예:
- `backups/firestore-backup-snapshot-YYYY-MM-DDTHH-mm-ss.json`

장점:
- 구현이 가장 단순
- 현재 작업 구조와 잘 맞음
- 바로 시작 가능

단점:
- 이 PC가 꺼져 있으면 백업이 안 됨
- 장기 보관/분산 보관은 약함

### 2단계. Cloud Storage 또는 별도 안전 보관소로 확장

추후에는
- 로컬 백업 생성 후
- 외부 저장소 또는 클라우드 저장소에도 복사합니다

장점:
- PC 고장/분실 시에도 상대적으로 안전

단점:
- 설정과 관리 포인트가 늘어남

## 5. 자동화 방식 후보

### 안 1. Windows 작업 스케줄러 기반

비유:
- 이 PC에 `정기 당직 직원`을 두는 방식

구성:
1. Node 스크립트 작성
2. 작업 스케줄러로 평일 정해진 시간 실행
3. 결과 JSON/로그 저장

장점:
- 지금 환경에서 가장 현실적
- 추가 인프라 거의 필요 없음

단점:
- PC가 켜져 있어야 함

### 안 2. Firebase/Cloud Functions 기반 배치

비유:
- 건물 안이 아니라 `본사 서버`에 야간 당직 직원을 두는 방식

구성:
1. Scheduled Function
2. Firestore 읽기
3. 백업 저장

장점:
- 로컬 PC에 의존하지 않음

단점:
- 구현 난이도 높음
- 운영/권한/과금 고려 필요

### 안 3. GitHub Actions 기반

비유:
- 설계도 창고 쪽 자동화 직원에게 장부 복사를 맡기는 방식

장점:
- 기록이 남기 쉬움

단점:
- Firestore 인증/비밀값 관리가 번거롭고
- 현재 구조에선 Windows 스케줄러보다 실익이 적음

## 6. 현재 추천안

**1차 추천: Windows 작업 스케줄러 + Node 백업 스크립트**

이유:
- 지금 프로젝트는 로컬 작업 기반이 강함
- 이미 서비스 계정 JSON과 복구 스크립트가 로컬에 있음
- 가장 적은 변경으로 빠르게 자동화 가능

즉:
- 먼저 `현장 당직 직원`부터 두고
- 나중에 필요하면 `본사 야간 배치`로 키우는 방식이 좋습니다

## 8. 백업 성공 기준

자동화가 성공했다고 보려면:
1. 매일 JSON 백업 파일 생성
2. 컬렉션 수량 요약 저장
3. 실패 시 히스토리에 기록
4. 최근 백업 파일이 실제로 열림
5. 복구 리허설 때 해당 백업을 참조 가능

## 9. 확장안

- 클라우드 드라이브
- Cloud Storage

## 10. 위험 요소

1. 서비스 계정 파일 경로가 바뀌면 백업 실패
2. PC가 꺼져 있으면 스케줄 실행 안 됨
3. 백업 파일만 계속 쌓이면 용량 증가
4. `accessLogs` 전체 포함 시 파일이 빨리 커질 수 있음
