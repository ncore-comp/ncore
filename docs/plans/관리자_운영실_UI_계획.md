# 관리자 운영실 UI 계획

## 1. 목적

현재 관리자 기능은 여러 곳에 흩어져 있다.

- 구성원 관리
- 특별휴가 설정
- 메일 라우트 설정
- 접속 로그 기록
- 연차 신청/삭제 관련 운영 업무
- 휴일 데이터 관리

운영 관점에서는 이것들을 한 화면에서 더 쉽게 다루는 것이 좋다.

쉽게 비유하면, 지금은 창고 안 여러 선반에 관리 도구가 흩어져 있고,  
앞으로는 `운영실` 한 곳에서 자주 쓰는 관리 도구를 한꺼번에 보게 만드는 작업이다.

## 2. 목표

관리자 전용 화면에 아래 3개 기능을 추가하거나 재배치한다.

1. 접속 로그 보기
2. 달력 기반 관리자 대리 등록/삭제
3. 수동 휴일 설정

중요:

- 자동 공휴일 데이터는 여기서 직접 수정하지 않는다
- 공공데이터 기반 자동 공휴일은 시스템이 관리한다
- 관리자 화면에서는 회사 휴일만 수동으로 다룬다

## 3. 현재 상태 분석

### 이미 있는 것

- 구성원 관리 화면 존재
- 특별휴가 종류 설정 화면 존재
- 사용자별 특별휴가 부여 화면 존재
- 메일 라우트 설정 화면 존재
- 접속 로그는 데이터로는 기록되고 있음

### 아직 약한 것

- 접속 로그를 관리자 UI에서 직접 보기 어렵다
- 연차 대리 등록/삭제를 운영자 관점으로 다루는 별도 UI가 없다
- 휴일은 현재 Firestore 직접 수정 비중이 남아 있다

## 4. 기본 방향

### 추천 방향

기존 `권한/설정` 화면을 확장해서 `운영실` 역할을 같이 하도록 만든다.

즉 새 건물을 짓지 않고, 현재 관리실 안에 아래 3개의 전용 구역을 만든다.

1. 접속 로그 구역
2. 달력 기반 대리 등록/삭제 구역
3. 수동 휴일관리 구역

### 이유

- 현재 구조를 크게 흔들지 않아도 된다
- 이미 관리자 설정 화면이 있으므로 자연스럽게 확장 가능하다
- 사용자가 이동할 페이지 수가 늘지 않는다

## 5. 화면 배치 제안

## 5-1. 상단 구조

현재 `권한/설정` 화면 구조를 유지하되, 하단에 운영 전용 섹션을 추가한다.

상단:

- 기존 권한/특별휴가/메일 라우트 설정

하단:

- 운영실 섹션

## 5-2. 운영실 섹션 구성

운영실 섹션은 3개의 카드 또는 탭으로 나눈다.

### A. 접속 로그

표시 목적:

- 누가
- 언제
- 어떤 동작을 했는지
를 빠르게 확인

표시 항목:

- 시간
- 이름
- 부서
- 동작 종류
- 상세 내용

추천 필터:

- 오늘
- 최근 7일
- 사용자별
- 로그 타입별

비유:

- 출입기록판

### B. 달력 기반 관리자 대리 등록/삭제

표시 목적:

- 관리자가 달력에서 다른 직원의 연차를 직접 등록
- 잘못 빠졌거나 누락된 요청을 달력에서 바로 보정
- 잘못 들어간 요청을 달력에서 바로 삭제

기능:

- 달력 날짜 클릭
- 직원 선택
- 연차/반차/시간차/특별휴가 선택
- 사유 입력
- 대리 등록
- 해당 날짜 기존 요청 목록 표시
- 선택 삭제

추가 원칙:

- 별도 폼 화면보다 `달력 위 보정 모드`가 중심
- Firestore 콘솔에서 직접 고치던 작업을 웹 안으로 가져오는 것이 목적
- 일반 신청 흐름을 새로 만드는 것이 아니라 운영자 전용 보정 흐름을 추가

비유:

- 달력 위에서 관리자가 빨간 펜으로 직접 누락분을 적고 지우는 보정 모드

### C. 수동 휴일 설정

표시 목적:

- 회사 창립기념일
- 여름휴가
- 임시휴무
같은 회사 전용 휴일만 관리

기능:

- 날짜 추가
- 휴일명 입력
- 활성/비활성
- 수정/삭제

중요:

- 자동 공휴일은 수정 대상에서 제외
- 필요하면 읽기 전용으로만 구분 표시

비유:

- 회사 전용 스티커 보드

## 6. 데이터 처리 원칙

### 6-1. 접속 로그

- 기존 `accessLogs` 컬렉션을 읽기 전용으로 사용
- 운영자 UI에서는 조회만 우선
- 삭제 기능은 당장 넣지 않는다

### 6-2. 달력 기반 관리자 대리 등록/삭제

- 기존 요청 저장 로직을 최대한 재사용
- 일반 직원과 같은 검증 규칙 사용
- 관리자만 다른 직원을 선택해 대리 등록 가능
- 대리 등록 여부를 메모 또는 로그에 남길 수 있게 하는 것이 좋다

### 6-3. 수동 휴일 설정

- `holidays` 컬렉션 사용
- `source = manual`만 수정 가능
- `source = auto`, `source = company_fixed`는 수정/삭제 불가 또는 읽기 전용

## 7. UI 규칙

1. 운영 데이터는 표 또는 카드형으로 즉시 읽히게 한다
2. 수정 가능 대상과 읽기 전용 대상을 명확히 색으로 구분한다
3. 삭제는 항상 확인창을 둔다
4. 자동 공휴일은 운영자가 실수로 지우지 못하게 한다

## 8. 구현 순서

1. 운영실 섹션 UI 골격 추가
2. 접속 로그 목록 표시 추가
3. 관리자 연차 대리 등록 UI 추가
4. 기존 요청 검색/삭제 UI 추가
5. 수동 휴일 목록/추가 UI 추가
6. 자동/회사고정/수동 휴일 구분 표시 추가
7. 최종 테스트

## 9. 테스트 계획

### 접속 로그

- 최근 로그가 화면에 보이는지
- 필터가 정상 동작하는지

### 달력 기반 관리자 대리 등록/삭제

- 관리자가 다른 직원을 선택해 달력에서 대리 등록 가능한지
- 등록 후 달력에 즉시 반영되는지
- 해당 날짜 요청 목록이 보이는지
- 삭제가 정상 작동하는지
- 삭제 확인이 뜨는지

### 수동 휴일 설정

- 회사 휴일 추가 가능
- 수정 가능
- 삭제 가능
- 달력에 반영
- 자동 공휴일은 수정/삭제 불가

## 10. 완료 기준

아래 조건을 만족하면 완료로 본다.

1. 접속 로그를 관리자 UI에서 확인할 수 있다
2. 관리자가 달력에서 다른 직원 연차를 대리 등록/삭제할 수 있다
3. 회사 휴일을 Firestore 콘솔 없이 웹에서 직접 관리할 수 있다
4. 자동 공휴일은 보호되고, 수동 휴일만 수정 가능하다

## 11. 한 줄 결론

이번 작업은 `권한/설정` 화면을 `운영실`까지 확장하되,  
연차 보정은 별도 입력 폼보다 `달력 위 대리 등록/삭제 모드`로 구현하는 것이 가장 자연스럽다.
