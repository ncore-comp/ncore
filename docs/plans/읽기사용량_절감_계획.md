# 읽기 사용량 절감 계획

## 1. 목적

- Firestore 읽기 사용량이 왜 커졌는지 정리한다.
- 당장 줄일 수 있는 조치를 우선 적용한다.
- 테스트 중 쌓인 `accessLogs`는 정리하고, 운영실 최근 로그 로드량을 줄인다.

쉽게 비유하면:
- 창고 문을 자주 여는 구간이 어디인지 먼저 찾고,
- 불필요하게 쌓인 출입장부를 치운 뒤,
- 경비실 모니터에는 최근 20장만 띄우도록 바꾸는 작업이다.

## 2. 읽기를 많이 먹는 항목 순위

### 1위. 백업 / 비교 / 점검 스크립트

이유:
- 컬렉션 전체를 통째로 읽는다.

대표 항목:
- `backup-firestore.js`
- `compare-firestore.js`
- 각종 검증 스크립트

특징:
- 한 번 실행할 때도 reads가 크게 뛴다.
- 그래프에서 특정 날짜에 튀는 큰 스파이크는 이 영향이 크다.

### 2위. 메인 데이터 로드

이유:
- 로그인/새로고침 시 `users`, `requests`, `boardPosts`, `holidays`, `specialLeaveTypes`, `userSpecialLeaves`, `mailRoutes`를 읽는다.

특징:
- 사용자가 여러 번 새로고침하거나 테스트를 반복하면 누적된다.

### 3위. 운영실 진입

이유:
- 운영실 로드 시 최근 접속 로그 + 휴일 데이터를 읽는다.

현재:
- 서버는 최근 로그 `150건`을 읽고 있음
- 화면은 그중 `20건`만 보여줌

즉:
- 보이는 것보다 더 많이 읽고 있는 상태다.

### 4위. 접속 로그 CSV 다운로드

이유:
- `accessLogs` 전체를 읽어서 CSV로 만든다.

현재 체감:
- 로그가 2371건이면
- 버튼 1번 = 대략 2371 reads 수준

특징:
- 누를 때마다 전체 장부 복사
- 자주 누르면 읽기 사용량이 빠르게 오른다.

### 5위. Firebase Console 직접 조회

이유:
- 콘솔에서 컬렉션/문서를 열어보는 것도 읽기로 잡힌다.

## 3. 지금 당장 줄일 수 있는 방법

### 1. 테스트용 accessLogs 삭제  
즉시 실행

배경:
- 현재 약 2371건의 접속 로그는 테스트 중 쌓인 기록으로, 운영상 필수 보관 대상이 아님

효과:
- 이후 CSV 다운로드 시 읽기량 감소
- 운영실 로그 관련 데이터량 감소
- 저장 용량도 감소

주의:
- 과거에 이미 사용된 읽기량 그래프는 줄어들지 않는다
- 앞으로의 읽기량만 줄어든다

### 2. 운영실 최근 로그 로드량을 20건으로 축소  
즉시 실행

현재:
- 서버 로드: 150건
- 화면 표시: 20건

변경:
- 서버 로드도 20건
- 화면 표시도 20건 유지

효과:
- 운영실 진입 때 불필요한 로그 읽기 감소

### 3. CSV 다운로드는 전체 유지  
유지

이유:
- 사용자가 전체 다운로드를 원함
- 다만 로그 수 자체를 줄이면 부담이 자동으로 줄어든다

## 4. 이번 실행 범위

이번에 바로 할 일:

1. `accessLogs` 전체 삭제
2. 운영실 최근 로그 로드량 `150 -> 20` 변경
3. 운영 서버 반영
4. 삭제 후 로그 수 확인
5. 문서 결과 기록

## 5. 이번에 하지 않을 것

1. CSV 다운로드 범위 제한
2. 메인 로드 구조 분해
3. 백업/비교 스크립트 구조 최적화

이유:
- 지금은 즉시 체감되는 조치부터 먼저 적용하는 것이 맞다.

아래가 되면 1차 절감 조치 완료로 본다.

3. 운영실 화면 정상 동작
4. 관련 문서 업데이트 완료
