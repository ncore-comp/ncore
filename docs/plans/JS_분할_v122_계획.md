## 왜 위험한가
## Why This Is Risky

- 지금 구조는 하나의 거대한 “본체” 파일에 상태, 유틸, 화면 렌더링, 이벤트 진입점이 전부 섞여 있습니다.
- The current structure mixes state, utilities, rendering, and entrypoints in one giant “main body” file.

- 그래서 잘못 나누면 아래 문제가 바로 생길 수 있습니다.
- A careless split can immediately cause the following issues.

1. `app`가 아직 생성되기 전에 분리 파일이 먼저 실행되는 문제
1. A split file executes before `app` exists

2. `app.init()` 시점이 앞당겨지거나 늦어져 초기 화면이 비정상인 문제
2. `app.init()` runs too early or too late and breaks initial rendering

3. 기능은 살아 있지만 특정 버튼이나 모달만 조용히 죽는 문제
3. Core pages appear fine but specific buttons or modals silently fail

4. 하나의 파일이 다른 기능군의 내부 헬퍼를 직접 참조하는 숨은 의존성 문제
4. Hidden cross-feature dependencies where one area directly uses another area’s internal helper

## 분할 원칙
## Split Principles

1. 한 번에 뒤집지 않습니다.
1. Do not flip everything at once.

- 한 단계에 한 기능군만 옮깁니다.
- Move only one feature group per step.

2. `core_logic`는 이번 1차에서 최대한 유지합니다.
2. Keep `core_logic` mostly unchanged in phase 1.

- 먼저 `ui_logic`만 쪼갭니다.
- Split `ui_logic` first.

3. `app` 본체를 먼저 “껍데기(shell)”로 분리합니다.
3. First separate the `app` body into a lightweight shell.

- 이후 기능 파일은 `Object.assign(app, { ... })` 형태로 붙입니다.
- Feature files should then attach methods via `Object.assign(app, { ... })`.

4. `app.init()`는 마지막 bootstrap 파일에서만 호출합니다.
4. Only call `app.init()` in the final bootstrap file.

5. 기능 이동 중에는 함수 내용 변경을 최소화합니다.
5. Minimize behavior changes while moving functions.

- 리팩터링보다 “이사”에 가깝게 진행합니다.
- Treat this more like relocation than refactoring.

6. 단계 사이에 반드시 “동작 동일성 게이트”를 둡니다.
6. Put a mandatory “behavior parity gate” between phases.

- 한 단계가 끝나면 다음 단계로 바로 넘어가지 않고, 현재 v122가 v121과 동일하게 동작하는지 먼저 확인합니다.
- After each phase, do not continue immediately; first verify that the current v122 still behaves the same as v121.

7. 공통 함수는 먼저 경계를 확정한 뒤에만 이동합니다.
7. Shared helpers may move only after their boundary is explicitly defined.

- “상황판 전용 함수”, “잔업/특근 전용 함수”, “여러 기능이 동시에 쓰는 공통 함수”를 구분하지 못하면 분할이 아니라 재결합이 됩니다.
- If we cannot distinguish “situation-only”, “workreport-only”, and “cross-feature shared” helpers, the split will collapse back into accidental coupling.

## 공통 헬퍼 경계표
## Shared Helper Boundary Table

### shell 에 남길 것
### Keep in Shell

- 전역 상태값
- `init`
- 화면 전환 진입점의 최소 껍데기
- `window.app = app`

### shared 로 뺄 것
### Move to Shared

- 날짜 포맷/시간 포맷
- 권한 판정 공통 함수
- 근무시간 계산 공통 함수
- 모달/알림 공통 렌더
- 요청 소유 판별, 상태 배지, 공통 텍스트 변환

### feature 로 옮길 것
### Move to Feature Modules

- 상황판 전용 렌더/팝업/패널
- 잔업/특근 전용 신청/현황/상세/정산
- 권한/설정/특별휴가/메일설정
- 구성원 관리/사용자 수정
- 달력 렌더/달력 상세/대시보드 묶음

### 이동 금지 기준
### “Do Not Move Yet” Rule

- 다른 기능군에서 2곳 이상 직접 호출하는 함수
- DOM id가 여러 화면에서 재사용되는 함수
- `app.init()` 직후 실행되는 초기화 함수

- 이런 함수는 먼저 `shared` 후보로 분류만 하고, 같은 단계에서 이름 변경까지 같이 하지 않습니다.
- These functions should first be classified as `shared` candidates; do not rename them in the same phase.
