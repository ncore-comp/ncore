## Goal

- For safety, do not delete immediately; instead follow `freeze production set -> move old versions to archive area -> consider final deletion`.

## 정리 원칙
## Cleanup Principles

1. 현재 운영본은 절대 즉시 삭제하지 않는다.
1. Never delete the current production files immediately.

2. 각 계열별로 `현재 운영본`과 `직전 기준본`은 함께 남긴다.
2. Keep both the current production file and the immediately previous baseline for each family.

3. 나머지 버전은 `js\\legacy_versions`로 먼저 이동한다.
3. Move the remaining versions to `js\\legacy_versions` first.

4. `ui_situation`처럼 다중 파일 로드 예외는 별도 표기로 관리한다.
4. Handle multi-file load exceptions like `ui_situation` explicitly.

5. 최종 삭제는 `운영 검증 1회 이상` 후에만 검토한다.
5. Consider final deletion only after at least one production verification round.

## 권장 구조
## Recommended Structure

### 유지
### Keep

- `js/` 루트:
  - 현재 운영본
  - 직전 기준본
  - 공통 무버전 파일

### 신설
### Create

- `js/legacy_versions/`
  - 더 이상 현재 작업 기준이 아닌 과거 버전 파일 보관
  - Store past version files that are no longer part of the current working baseline

- `docs/reference/현재운영_JS버전맵_YYYY-MM-DD.md`
  - 현재 운영 파일 목록과 예외 조합 기록
  - Record current production file list and exception combinations

## 단계별 계획
## Step-by-Step Plan

### 1단계
### Phase 1

- 현재 운영 기준 파일 목록을 문서로 고정
- Freeze the current production file list in a document

- 각 계열별 `운영본 / 직전본 / legacy 대상`을 분류
- Classify each family into `production / previous / legacy`

### 2단계
### Phase 2

- `js\\legacy_versions` 생성
- Create `js\\legacy_versions`

- 운영본이 아닌 오래된 버전 파일을 계열별 하위 폴더 또는 flat 구조로 이동
- Move non-production old version files into `legacy_versions`

- `index.html` 참조에는 영향이 없도록 유지
- Keep `index.html` references unchanged

### 3단계
### Phase 3

- 문서/스크립트 중 옛 파일을 직접 가리키는 경우 링크 경로 정리
- Clean up any docs/scripts that directly reference old file paths

- 필요 시 최종 삭제 후보 목록 작성
- If needed, prepare a final deletion candidate list
