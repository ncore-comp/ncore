# 특별휴가 발생일·기한·경고 계획
# Special Leave Event-Date, Expiry, and Warning Plan

## 1. 목적
## 1. Goal

- 특별휴가를 발생일 기준 규칙에 따라 제한적으로 사용하게 만든다.
- Make special leave usable only under event-date-based rules.

- 다 썼거나 기한이 지난 특별휴가는 연차현황표에서 숨긴다.
- Hide fully used or expired special leave from the annual status card.

- 미사용 특별휴가는 남은 사용 가능 평일 수와 남은 일수를 비교해 경고한다.
- Warn about unused special leave by comparing remaining usable weekdays and remaining leave days.

- 최종적으로 미사용 상태로 기한을 넘기면 소멸 팝업으로 안내한다.
- If unused leave passes its deadline, show an expiry popup.

- Firestore 읽기/쓰기는 마지막 단계 전까지 하지 않거나 최소화한다.
- Avoid or minimize Firestore reads/writes until the final stage.

## 2. 적용 범위
## 2. Scope

- 본 계획은 `경조휴가성 특별휴가`를 대상으로 한다.
- This plan covers ceremonial or discretionary special leave.

- `출산휴가`는 본 계획 대상에서 제외한다.
- `Maternity leave` is excluded from this plan.

- 출산휴가는 법정 규칙을 따라야 하므로 별도 문서에서 다룬다.
- Maternity leave must follow statutory rules and should be handled in a separate document.

## 3. 기본 정책
## 3. Base Policy

- 사용 가능 기간 계산은 `주말만 제외`한다.
- Expiry-window calculations exclude weekends only.

- 회사 휴일 / 공휴일은 평일이면 포함한다.
- Company holidays / public holidays are counted if they fall on weekdays.

- 발생 당일은 사용 가능 기간 계산에서 `포함하지 않는다`.
- The event date itself is `excluded` from the usable-window calculation.

- 소멸기한 규칙은 특별휴가 `종류`에 붙인다.
- Expiry rules are attached to the special leave `type`.

- 특별휴가는 경조사 특성상 같은 종류도 한 해에 여러 건 발생할 수 있다.
- Because special leave often comes from ceremonial events, the same leave type may occur multiple times in one year.

- 따라서 사용자별 특별휴가는 `종류별 1건`이 아니라 `부여 건별(grant 단위)`로 관리한다.
- Therefore, user special leave must be managed per `grant instance`, not as a single record per leave type.

- 실제 사용 가능 기간은 `종류 규칙 + 사용자별 발생일`을 조합해 계산한다.
- The actual usable window is computed from `type rule + user-level event date`.

## 4. 현재 구조 요약
## 4. Current Structure Summary

- 특별휴가 종류 설정은 현재 `이름`, `기본 일수`, `사용 타입`, `차감 방식`, `주말/공휴일 허용` 중심이다.
- Special leave types currently store `label`, `grant days`, `request mode`, `day count mode`, and `holiday allowance`.

- 사용자별 특별휴가 데이터는 현재 `totalHours`, `usedHours`, `note`, `updatedAt` 정도만 저장한다.
- User-level special leave grant data currently stores `totalHours`, `usedHours`, `note`, and `updatedAt`.

- 연차현황표와 신청 가능 목록은 현재 `남은 시간`만 보고 특별휴가를 노출한다.
- The annual status card and request option list currently expose special leave based only on remaining hours.

## 5. 목표 워크플로우
## 5. Target Workflow

### 5-1. 특별휴가 종류 등록
### 5-1. Special Leave Type Registration

- 특별휴가 종류 설정 화면에 `발생일 기준` 안내 문구를 고정 표시한다.
- Show a fixed `발생일 기준 / Based on event date` label in the special leave type settings UI.

- 차감 방식이 `영업일 기준(business_days)`일 때만 소멸기한 설정 입력을 활성화한다.
- Enable expiry-rule inputs only when `dayCountMode = business_days`.

- 입력 항목
- Input fields

- `소멸 방향(expiryDirection)` 드롭다운
- `Expiry direction` dropdown

  - `전(before)`
  - `후(after)`
  - `전후(both)`

- `기한 일수(expiryDays)` 수동 숫자 입력
- `Expiry days` manual numeric input

- `전후 + 10일`은 `앞 10일 + 뒤 10일`을 뜻한다.
- `both + 10 days` means `10 weekdays before + 10 weekdays after`.

- 발생 당일은 포함하지 않는다.
- Exclude the event date itself.

- 차감 방식이 `달력일 기준(calendar_days)`이면
- When `dayCountMode = calendar_days`

  - 소멸 방향 드롭다운 비활성화
  - Disable the expiry-direction dropdown

  - 기한 일수 입력 비활성화
  - Disable the expiry-days input

  - 저장 시 관련 값은 비우거나 무시
  - Clear or ignore those values on save

- `day_only` 특별휴가는 1일 단위만 허용한다.
- `day_only` special leave must be used in whole-day units only.

### 5-2. 사용자별 특별휴가 부여
### 5-2. User-Level Special Leave Grant

- 관리자는 `구성원 관리 > 직원 수정 > 특별휴가`에서 부여한다.
- Admins grant special leave from `User Management > Edit User > Special Leave`.

- 기존의 `종류별 체크박스 + 총량 입력` 중심 UI는 `grant 행 추가` 중심 UI로 바꾼다.
- Replace the current `per-type checkbox + total amount` UI with an `add grant row` workflow.

- 각 grant 행에는 최소한 아래 입력이 있어야 한다.
- Each grant row must include at least the following inputs.

  - 특별휴가 종류
  - Special leave type

  - 발생일
  - Event date

  - 부여 일수
  - Granted days

  - 수동 사용 일수(필요 시)
  - Manually used days if needed

- `발생일`은 필수 입력이다.
- `Event date` is required.

- 발생일이 비어 있으면 저장할 수 없다.
- Saving must be blocked if the event date is empty.

- 같은 종류라도 경조사가 다시 발생하면 새 `grant`를 추가한다.
- Even for the same leave type, create a new `grant` when a new event occurs.

- 저장 전 화면에서 `사용 가능 기간 미리보기`를 보여준다.
- Show a `usable window preview` before saving.

- `usableFromDate`, `usableToDate`는 화면 계산값으로 먼저 만들고, 최종 단계에서만 저장 여부를 결정한다.
- Compute `usableFromDate` and `usableToDate` in the UI first, and decide whether to persist them only in the final stage.

- 요청이 한 번도 연결되지 않은 grant만 자유 수정/삭제한다.
- Allow free edit/delete only for grants that have no linked requests.

- 한 번이라도 요청이 연결된 grant는 자유 수정/삭제를 금지한다.
- Once any request has been linked to a grant, block free edit/delete.

### 5-3. 특별휴가 신청
### 5-3. Special Leave Request

- 특별휴가를 부여받은 직원이 연차 신청 화면을 열면, 특별휴가는 `부여 건별`로 각각 표시한다.
- When an employee opens the leave request form, special leave must appear per `grant instance`.

- 요청 종류 표시는 `종류 / 발생일`로 구분한다.
- Distinguish request type options by `type / event date`.

- 예시
- Example

- `특별휴가(결혼(본인) / 발생일 2026-04-08)`
- `Special Leave (Marriage(Self) / Event Date 2026-04-08)`

- `특별휴가(결혼(본인) / 발생일 2026-10-21)`
- `Special Leave (Marriage(Self) / Event Date 2026-10-21)`

- 신청 드롭다운 정렬은 `usableToDate 빠른 순 -> eventDate 빠른 순`으로 한다.
- Sort the request dropdown by `earliest usableToDate -> earliest eventDate`.

- 사용자가 선택한 특별휴가 옵션은 `specialLeaveGrantId`로 요청에 저장한다.
- Save the selected special leave option on the request as `specialLeaveGrantId`.

- 차감되는 특별휴가는 사용자가 고른 grant와 1:1로 연결한다.
- Deduct the exact grant selected by the user on a 1:1 basis.

- 선택한 grant의 잔여가 부족하면 저장을 차단한다.
- Block saving when the selected grant does not have enough remaining balance.

- 여러 grant를 한 요청에서 자동 합산 차감하지 않는다.
- Do not automatically split or combine multiple grants for a single request.

- 특별휴가가 여러 건 부여됐으면 각 grant를 독립적으로 사용해야 한다.
- If multiple special-leave grants exist, each grant must be used independently.

### 5-4. 승인 / 반려 / 취소
### 5-4. Approve / Reject / Cancel

- 승인/반려/취소 흐름은 일반 연차와 동일하게 처리한다.
- Handle approve/reject/cancel the same way as regular annual leave.

- `pending`일 때는 예약 상태로 본다.
- Treat `pending` as reserved.

- `approved`가 되면 실제 사용량으로 반영한다.
- Convert to actual used hours on `approved`.

- `rejected`, `cancelled`가 되면 예약을 해제한다.
- Release the reservation on `rejected` or `cancelled`.

- 신청 시점 기준으로 유효하면 승인 시점에 기한이 지나도 승인 가능하다.
- If a request was valid at submission time, it can still be approved even if the expiry date has passed by approval time.

### 5-5. 사용자 로그인 시 경고
### 5-5. Login-Time Warning

- 로그인할 때마다 세션당 1회만 팝업을 띄운다.
- Show one popup per login session.

- 팝업은 여러 건을 묶어서 한 번에 보여준다.
- Aggregate multiple items into a single popup.

- 팝업 안에는 `소멸 n건 / 임박 n건` 요약을 함께 표시한다.
- Show combined counts such as `expired n items / near-expiry n items` in the popup.

- 팝업 확인 후 안내 문구는 `달력에서 확인하세요`로 한다.
- After acknowledging the popup, show guidance text such as `달력에서 확인하세요 / Please check the calendar`.

- 추후 달력에 특별휴가 발생일과 사용 가능 기간을 시각적으로 표시할 계획을 둔다.
- Plan to visually surface special-leave event dates and usable windows on the calendar later.

## 6. 목표 동작
## 6. Target Behavior

- 특별휴가 종류에 소멸 규칙이 있고, 사용자 부여 데이터에 발생일이 있으면 사용 가능 기간을 계산한다.
- If a special leave type has an expiry rule and the user grant has an event date, compute its usable window.

- `availableHours <= 0` 이면 연차현황표에서 숨긴다.
- Hide the entry from the annual status card when `availableHours <= 0`.

- `today > usableToDate` 이고 `availableHours > 0` 이면 숨기고 소멸 팝업 대상에 넣는다.
- If `today > usableToDate` and `availableHours > 0`, hide it and include it in the expiry-popup list.

- `availableHours = grantedHours - approvedUsedHours - pendingHours` 로 계산한다.
- Compute `availableHours = grantedHours - approvedUsedHours - pendingHours`.

- 여기서 `pendingHours`는 같은 `grantId`에 연결된 `pending` 요청 합계다.
- Here, `pendingHours` means the sum of `pending` requests linked to the same `grantId`.

- `availableHours > 0` 이면 `requiredUseDays = max(1, ceil(availableHours / 8))`를 계산한다.
- If `availableHours > 0`, compute `requiredUseDays = max(1, ceil(availableHours / 8))`.

- `remainingWeekdaysUntilExpiry <= requiredUseDays` 이면 로그인 경고 팝업 대상에 넣는다.
- If `remainingWeekdaysUntilExpiry <= requiredUseDays`, include it in the login warning-popup list.

- 사용자 표시 문구는 `남은 n일` 기준으로 보여준다.
- Show user-facing messages in `remaining n days` form.

- 내부 경고 판정은 시간 기반이어도, 사용자 문구는 `남은 1일`, `남은 1일 4시간`처럼 읽기 쉬운 표현으로 노출한다.
- Even if the internal warning logic is time-based, user-facing copy should use readable forms like `1 day left` or `1 day 4 hours left`.

- 특별휴가 신청은 `usableFromDate ~ usableToDate` 범위 안에서만 가능하다.
- Allow special leave requests only within `usableFromDate ~ usableToDate`.
