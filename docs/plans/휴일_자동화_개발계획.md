# 휴일 자동화 개발계획

## 1. 목적

현재 달력의 휴일 데이터는 수동 입력만으로 관리한다.

원하는 최종 상태는 아래와 같다.

- 대한민국 기본 공휴일, 명절, 대체공휴일은 자동으로 반영
- 회사 창립기념일, 여름휴가, 임시휴무일은 수동으로 반영
- 달력 화면에서는 자동/수동을 구분하지 않고 하나의 휴일 데이터처럼 사용

쉽게 비유하면 아래와 같다.

- 기본 달력 인쇄판: 대한민국 공휴일 자동 데이터
- 회사 스티커: 회사만의 휴일 수동 데이터
- 최종 달력: 인쇄판 위에 스티커를 붙여 완성된 달력

## 2. 구현 원칙

이번 구조는 아래 원칙으로 간다.

1. 회사 휴일을 직접 넣는 현재 수동 방식은 유지한다.
2. 대한민국 공휴일은 신뢰 가능한 외부 데이터로 자동화한다.
3. 자동 데이터와 수동 데이터가 겹치면 수동 데이터가 우선한다.
4. 프론트는 최대한 건드리지 않고 서버 또는 데이터 준비 단계에서 합쳐서 내려준다.
5. 운영자가 매년 달력을 손으로 다시 만들지 않게 한다.

## 3. 자동 데이터 소스 방향

### 추천 방향

- 대한민국 공휴일 자동 데이터는 `공공데이터포털 / 한국천문연구원 특일 정보` 계열을 우선 검토한다.

이유:

- 대한민국 공휴일/명절 기준 데이터로 신뢰도가 높다.
- 네이버/구글 화면용 캘린더보다 시스템 백엔드 데이터 소스로 더 적합하다.
- 내부 계산 로직으로 설날, 추석, 대체공휴일을 직접 구현하는 것보다 유지보수가 쉽다.

### 비추천 방향

- 설날/추석/대체공휴일 계산 로직을 내부에서 직접 구현

이유:

- 자동처럼 보이지만, 규칙이 바뀌면 결국 개발자가 매번 관리해야 한다.
- 사용자 관점에서는 사실상 수동 유지보수와 다를 바 없다.

## 4. 최종 구조

최종 구조는 아래 3층으로 나눈다.

### 4-1. 자동 공휴일 데이터

- 대한민국 공휴일
- 명절
- 대체공휴일

이 데이터는 연도별로 자동 수집 또는 갱신한다.

### 4-2. 수동 회사 휴일 데이터

- 창립기념일
- 여름휴가
- 임시휴무
- 사내 행사 휴무

이 데이터는 운영자가 직접 등록한다.

### 4-3. 병합된 최종 휴일 데이터

화면에는 최종적으로 하나의 휴일 맵만 전달한다.

예시:

```text
2027-01-01 -> 신정
2027-03-01 -> 삼일절
2027-07-29 -> 여름휴가
```

## 5. 우선순위 규칙

같은 날짜에 자동 데이터와 수동 데이터가 동시에 있으면 아래 규칙을 따른다.

1. 수동 데이터 우선
2. 자동 데이터는 기본값 역할

예시:

- 자동: `2027-08-15 -> 광복절`
- 수동: `2027-08-15 -> 광복절 휴무`
- 최종 표시: `광복절 휴무`

이 규칙을 쓰면 회사가 표기명을 조금 다르게 쓰고 싶을 때도 대응 가능하다.

## 6. 데이터 형식 기준

수동 입력 데이터는 아래 형식을 강제한다.

- `date`: `YYYY-MM-DD`
- `name`: 휴일명
- `source`: `manual`
- `enabled`: `true/false`

자동 생성 데이터는 아래 형식으로 맞춘다.

- `date`: `YYYY-MM-DD`
- `name`: 휴일명
- `source`: `auto`
- `enabled`: `true`

중요:

- 값 안에 따옴표를 직접 넣지 않는다.
- 예: `2027-01-01`
- 잘못된 예: `"2027-01-01"`

## 7. 구현 방식

### 7-1. 1단계: 자동 데이터 수집기 만들기

할 일:

- 연도 기준으로 대한민국 공휴일 데이터를 가져오는 스크립트 작성
- 가져온 데이터를 내부 표준 형식으로 변환

결과:

- `year -> holiday list` 형태 생성

### 7-2. 2단계: 수동 데이터 읽기 유지

할 일:

- 현재 운영자가 직접 넣는 휴일 데이터 읽기 구조 유지
- 기존 수동 등록 방식은 없애지 않는다.

결과:

- 현재 운영 흐름을 깨지 않는다.

### 7-3. 3단계: 자동 + 수동 병합 로직 작성

할 일:

- 자동 데이터 먼저 생성
- 수동 데이터를 읽어서 같은 날짜를 덮어쓰기
- 최종 `holidays` 맵 생성

결과:

- 프론트는 지금처럼 하나의 `holidays`만 사용하면 된다.

### 7-4. 4단계: 화면 반영 확인

할 일:

- 월간 달력에서 휴일명 노출 확인
- 공휴일 날짜색 반영 확인
- 신청 가능 여부 규칙과 충돌 없는지 확인

## 8. 운영 방식

운영은 아래처럼 매우 단순해야 한다.

### 운영자가 해야 하는 일

- 회사 휴일만 수동 입력
- 자동 공휴일 데이터는 신경 쓰지 않음

### 시스템이 해야 하는 일

- 연도별 대한민국 공휴일 가져오기
- 수동 데이터와 자동 병합
- 달력에 반영

즉 운영자는 `회사 스티커만 붙이고`, 시스템이 `기본 인쇄 달력`을 준비하는 구조다.

## 9. 구현 후보 방식 비교

### 안 1. 접속 시마다 외부 데이터 호출

장점:

- 항상 최신

단점:

- 외부 장애에 직접 영향받음
- 속도 느려질 수 있음
- API 제한/키 관리 필요

판단:

- 비추천

### 안 2. 연도별 자동 동기화 후 저장

장점:

- 속도 안정적
- 외부 API 장애가 있어도 저장된 데이터 사용 가능
- 운영자가 보기에도 단순함

단점:

- 동기화 시점을 정해야 함

판단:

- 추천

### 안 3. 내부 계산 로직 구현

장점:

- 외부 의존 없음

단점:

- 유지보수 부담 큼
- 진짜 자동이라고 보기 어려움

판단:

- 비추천

## 10. 최종 추천안

최종 추천안은 아래와 같다.

1. 대한민국 공휴일은 공공데이터 계열 소스에서 연도별 자동 동기화
2. 회사 휴일은 지금처럼 수동 등록 유지
3. 서버에서 자동 + 수동 병합
4. 화면은 병합된 결과만 사용

이 방식의 장점:

- 운영자는 회사 휴일만 관리하면 됨
- 대한민국 공휴일은 매년 반복 수작업이 줄어듦
- 화면 구조를 크게 바꾸지 않아도 됨

## 11. 개발 순서

1. 외부 데이터 소스 최종 확정
2. 연도별 자동 공휴일 수집 스크립트 작성
3. 수동 휴일 데이터 형식 정리
4. 병합 함수 작성
5. 달력 화면 반영 테스트
6. 관리자 운영 가이드 문서 작성

## 12. 테스트 계획

아래 테스트를 통과해야 한다.

1. 대한민국 기본 공휴일이 자동으로 뜬다.
2. 설날, 추석, 대체공휴일이 올바르게 뜬다.
3. 회사 창립기념일을 수동 입력하면 달력에 뜬다.
4. 같은 날짜에 자동/수동이 겹치면 수동 이름이 보인다.
5. 월간 달력 날짜 색상이 휴일 기준으로 정상 반영된다.
6. 새로고침/재로그인 후에도 유지된다.
7. 다음 연도 달력도 자동 공휴일이 표시된다.

## 13. 운영상 주의점

1. 수동 데이터 필드 값에 따옴표를 직접 넣지 않는다.
2. 자동 데이터가 잘 들어왔는지 연초에 한 번 점검한다.
3. 회사 휴일은 자동 데이터가 아니라 수동 입력 대상임을 명확히 한다.
4. 외부 소스 장애 시 마지막 정상 데이터로 유지되도록 설계한다.

## 14. 완료 기준

아래 조건을 만족하면 완료로 본다.

- 대한민국 공휴일이 자동 반영된다.
- 회사 휴일을 수동으로 추가할 수 있다.
- 두 데이터가 충돌 없이 병합된다.
- 운영자가 해마다 명절/대체공휴일을 손으로 넣지 않아도 된다.
- 테스트 항목이 모두 통과한다.

## 15. 한 줄 결론

이번 휴일 자동화는 `대한민국 공휴일은 자동 인쇄`, `회사 휴일은 수동 스티커`, `서버가 둘을 합쳐 최종 달력으로 보여주는 구조`로 가는 것이 가장 안정적이다.
