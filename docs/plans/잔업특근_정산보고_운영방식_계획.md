# 잔업특근 정산보고 운영방식 계획

- 작성일시: 2026-04-01 12:37:00
- 대상 기능: 잔업 보고, 특근 보고, 정산 메일 제출 흐름

## 1. 배경

- 잔업/특근은 현재 시스템에 `건별 보고형 데이터`로 기록할 수 있다.
- 실제 운영 방식은 첨부 예시처럼 `한 달치 보고서 1통`으로 묶어서 보내되,
  표 안의 각 행은 날짜별 실제 수행 내용을 그대로 담아야 한다.
- 즉, `업무내용`, `요청 부서`, `비고`는 날짜마다 달라질 수 있으며,
  단순 총합 보고가 아니라 `건별 상세를 유지한 정산 보고`가 필요하다.

## 2. 기본 원칙

### 2.1 기록과 메일 보고를 분리한다

- 잔업/특근 사용자는 사용한 날 또는 직후에 시스템에 `건별 기록`만 한다.
- 메일 보고는 매번 보내지 않고, 정산일에 한 번에 묶어서 제출한다.

쉽게 말하면:
- 시스템 입력 = 가계부처럼 그때그때 적는 기록
- 메일 제출 = 월말 정산서처럼 한 번에 보내는 보고

### 2.2 정산 기준 기간

- 정산기간은 `매월 16일 ~ 다음달 15일`
- 예:
  - `2026년 3월 정산 = 2026-02-16 ~ 2026-03-15`

### 2.3 메일 제출 기준일

- 기본 제출일: 매월 `16일`
- 단, 16일이 휴일이면 `차주 월요일`에 제출

## 3. 추천 운영 방식

### 3.1 사용자 운영 흐름

1. 잔업/특근 발생
2. 사용자는 시스템에 `건별 기록`
3. 상태는 우선 `기록됨`으로 둔다
4. 정산일이 되면 시스템이 해당 정산기간의 미제출 건을 취합
5. 사용자는 정산 메일 초안을 검토 후 `제출`
6. 제출 완료 시 상태는 `정산 제출 완료`

### 3.2 관리자 운영 흐름

- 관리자는 매건 승인하지 않는다
- 대신 아래를 확인한다
  - 정산기간별 제출 여부
  - 사용자별 총 잔업시간
  - 사용자별 총 특근일수
  - 상세내역
- 미제출자만 별도로 확인하면 된다

## 4. 보고서 구조 원칙

### 4.1 건별 행(row)을 유지한다

정산 메일은 단순히 총합만 보여주는 방식이 아니라,
첨부 예시처럼 `날짜별 행`을 유지한 표여야 한다.

각 행에는 최소한 아래 항목이 들어가야 한다.

- 잔업일자
- 작업시작~완료
- 작업시간
- 잔업/특근
- 업무내용
- 요청 부서
- 비고

### 4.2 날짜별 내용은 서로 달라도 된다

중요한 점:

- `업무내용`은 날짜마다 다를 수 있다
- `요청 부서`도 날짜마다 다를 수 있다
- `비고`도 날짜마다 다를 수 있다

즉 하나의 정산 메일 안에서도,
각 날짜 행은 서로 다른 설명을 가져야 한다.

### 4.3 합계는 맨 아래에만 둔다

표 맨 아래에는 합계를 둔다.

- 총 잔업시간
- 총 특근일수

하지만 합계 때문에 각 날짜 행의 개별 정보가 합쳐지거나 생략되면 안 된다.

한 줄 요약:
- `행별 상세는 그대로 유지`
- `합계는 맨 아래에 별도로 표시`

## 5. 권장 상태값

현재 승인형 상태를 그대로 쓰기보다, 잔업/특근은 보고형 상태로 분리하는 것이 적절하다.

### 권장 상태

- `reported_draft`
  - 건별 기록만 된 상태
  - 정산 메일 제출 전
- `reported_submitted`
  - 정산 메일 제출 완료

쉽게 말하면:
- `입력은 했는가`
- `이번 달 보고까지 끝냈는가`
만 보면 된다.

## 6. 메일 보고 방식

### 6.1 건별 메일은 비추천

- 잔업/특근 등록 때마다 메일을 보내면
  - 메일이 너무 많아지고
  - 관리자가 정산용으로 다시 모아야 한다

### 6.2 정산 메일 1통 방식 추천

#### 메일 제목 예시

- `[잔업/특근 보고] 최은지 / 2026년 3월 정산`

#### 메일 본문 구성 추천

- 인사 문구
- 정산기간:
  - `2026-02-16 ~ 2026-03-15`
- 표:
  - 날짜
  - 작업시작~완료
  - 작업시간
  - 구분(잔업/특근)
  - 업무내용
  - 요청 부서
  - 비고
- 합계:
  - 총 잔업시간
  - 총 특근일수

## 7. 시스템 구현 추천 방향

### 7.1 현재 구조 재사용

- 기존 `requests` 컬렉션과 `reported` 계열 구조를 최대한 재사용
- 잔업/특근은 계속 `건별 기록형 데이터`로 저장

### 7.2 추가되면 좋은 기능

1. 정산 제출 배너
- 매월 16일(휴일이면 차주 월요일)에
- `이번 정산기간 미제출 잔업/특근 n건` 안내

2. 메일 초안 자동 생성
- 사용자가 `제출` 버튼을 누르면
- 해당 정산기간 데이터를 자동으로 표 형태로 묶어 메일 본문 생성

3. 제출 완료 표시
- 사용자별 `이번 정산 제출 완료 / 미제출` 상태 확인

4. 관리자 미제출 확인
- 관리자 화면에서 `미제출자`만 필터링

## 8. 예외 처리 기준

### 8.1 제출일이 휴일인 경우

- 16일이 토/일/공휴일이면
- `차주 월요일` 제출로 자동 이월

### 8.2 정산기간 외 데이터

- 제출 시점에는 `선택한 정산기간`에 해당하는 건만 포함
- 이전 기간 제출 누락건은 별도 구분 또는 관리자 확인 필요

### 8.3 사용자는 기록을 미리 쌓을 수 있다

- 선사용 후보고 구조이므로
- 메일은 나중에 보내더라도 기록 자체는 미리 쌓여 있어야 한다

## 9. 추천안 결론

가장 좋은 운영 방식은 아래와 같다.

- 잔업/특근은 발생할 때마다 시스템에 `건별 기록`
- 메일은 `매월 16일(휴일이면 차주 월요일)`에 정산기간 단위로 `1회 제출`
- 메일 표는 `건별 날짜 행`을 그대로 유지
- 합계는 표 맨 아래에서 별도로 표시
- 관리자는 승인자가 아니라 `정산 제출 확인자` 역할

한 줄 요약:
- `건별 기록 + 정산일 일괄 제출 + 날짜별 상세 유지`가 가장 자연스럽고, 현재 실제 보고 방식과도 가장 잘 맞는다.
