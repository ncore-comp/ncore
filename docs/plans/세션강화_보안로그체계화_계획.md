# 세션 강화 / 보안 로그 체계화 계획

## 1. 목적

- 현재 로그인/세션 구조를 한 단계 더 강화한다.
- 보안 이벤트를 일반 접속 로그와 구분해서 남길 수 있는 기준을 만든다.
- 이미 끝난 `비밀번호 / 로그인 제한 / 권한 분리 / load_data 최소화` 다음 단계로 이어지는 보안 하드닝 계획을 정리한다.

쉽게 말하면:
- 지금은 `출입카드`는 잘 발급되고 있지만,
- 앞으로는 `다른 사람이 그 카드를 들고 들어오는지`도 더 잘 보고,
- `누가 문을 몇 번 따보려 했는지`를 경비실 장부에 따로 남기는 단계다.

## 2. 현재 상태 요약

이미 완료된 것:
- 비밀번호 해시 전환
- 비밀번호 변경 재인증
- 로그인 5회 실패 / 5분 차단
- 승인권한과 운영실 수정권한 분리
- `load_data` 1차 / 2차 축소
- 게시판 / 전체 상황판 전용 로드 분리

현재 남은 대표적인 보안 고도화 과제:
- 세션 유출/재사용 방어 강화
- 비정상 로그인/권한 변경/운영실 작업을 보안 이벤트로 따로 기록

## 3. 세션 강화 계획

### 3.1 현재 구조

- 세션 토큰은 서버에서 발급
- 토큰 원문은 클라이언트가 보관
- 서버는 해시된 토큰으로 검증
- `createdIp`, `lastSeenAt`, `expiresAt`는 저장됨

현재 장점:
- 세션 없는 읽기/쓰기 차단 완료
- 토큰 원문을 Firestore에 그대로 저장하지 않음

현재 약점:
- 세션 생성 시 저장한 `createdIp`를 실제 검증에 적극적으로 쓰지 않음
- User-Agent 기반 비교가 없음
- 세션 재사용 이상징후를 경고/차단하는 단계가 없음

### 3.2 목표

1. `세션 탈취 재사용` 위험을 줄인다
2. 정상 사용자 불편은 최소화한다
3. 테스트 기간과 운영 기간을 나눠서 적용 가능하게 한다

### 3.3 추천 방향

#### 1안. 추천

- 세션 생성 시 `createdIp` + `createdUserAgentHash` 저장
- 요청마다 현재 IP / UA를 비교
- 완전 차단이 아니라 `위험도 기반 처리`

예:
- 같은 UA, 비슷한 IP 대역: 통과
- 완전히 다른 UA, 다른 환경: 보안 이벤트 기록
- 정책에 따라 추후 강제 로그아웃 또는 재인증 확장 가능

장점:
- 지금 운영 흐름을 크게 안 깨뜨림
- 보안 이벤트 기반 확장 가능

#### 2안. 강경안

- 세션 생성 당시 IP/UA가 조금이라도 다르면 즉시 세션 무효화

장점:
- 가장 강함

단점:
- 모바일망 / 사내망 / 브라우저 업데이트 등에서 정상 사용자도 자주 튕길 수 있음

현재 추천:
- 1안

### 3.4 실제 구현 순서

1. 세션 문서에 `createdUserAgentHash`, `lastSeenIp`, `lastSeenUserAgentHash` 추가
2. 요청 시 현재 IP / UA 비교 유틸 추가
3. 차이가 크면 `security log` 기록
4. 운영 정책 결정 후 필요 시 차단/재인증 단계 추가

## 4. 보안 로그 체계화 계획

### 4.1 현재 구조

- 현재는 `accessLogs` 중심
- 로그인/로그아웃/일반 작업 로그는 남지만
- 보안 관점에서 따로 묶여 있지 않음

쉽게 말하면:
- 지금은 `출입기록`만 있고
- `문 강제 개방 시도`, `비밀번호 연속 실패`, `권한 이상 변경` 같은 사건일지는 따로 없는 상태다.

### 4.2 목표

- 보안 이벤트를 일반 운영 로그와 분리해서 본다
- 나중에 관리자 화면 / CSV / 알림으로 확장 가능하게 만든다

### 4.3 보안 로그로 남길 이벤트

우선 1차 후보:
- 로그인 실패
- 로그인 차단 발생
- 비밀번호 변경 성공
- 비밀번호 변경 실패
- 세션 이상징후 감지
- 권한 저장/변경
- 운영실 대리 수정/삭제
- 공휴일 동기화 실행
- 전체 로그 CSV 다운로드

### 4.4 저장 방식 추천

#### 1안. 추천

- 새 컬렉션 `securityLogs` 추가
- 일반 `accessLogs`와 분리

필드 예:
- `timestamp`
- `sortTimestamp`
- `userId`
- `userName`
- `eventType`
- `severity`
- `ip`
- `userAgentHash`
- `detail`

장점:
- 일반 접속 로그와 안 섞임
- 조회/필터/CSV가 쉬움

#### 2안. 대안

- `accessLogs`에 `category=security`만 추가

장점:
- 구현이 빠름

단점:
- 운영 로그와 보안 로그가 섞여서 나중에 관리가 어려움

현재 추천:
- 1안

## 6. 완료 기준

세션 강화 완료 기준:
- 세션에 IP / UA 비교 정보가 남아 있음
- 이상 세션 사용 시 보안 로그가 남음

보안 로그 완료 기준:
- 일반 접속 로그와 별도로 보안 이벤트 기록 가능
- 로그인 실패/차단/비밀번호 변경/권한 변경/운영실 주요 작업이 구분 기록됨

## 7. 추천 우선순위

1. 세션 메타 확장
2. `securityLogs` 컬렉션 도입
3. 로그인 / 비밀번호 / 권한 변경 로그 분리
4. 운영실 조회 화면 확장 여부 판단

한 줄 정리:
- 다음 보안 단계는 `세션을 더 똑똑하게 확인`하고, `보안 사건만 따로 기록하는 장부`를 만드는 작업이다.
