## 1. 목적

이번 리허설의 목적은
- 현재 Firestore 운영 데이터를 안전하게 백업할 수 있는지
- 엑셀 원본과 import 스크립트로 복구가 가능한지
- 복구 후 핵심 서비스가 정상 동작하는지
를 실제 절차로 확인하는 것입니다.

쉽게 비유하면:
- Firestore는 `현재 운영 장부`
- `Ncore_DB.xlsx`는 `백업 장부`
- 이번 리허설은 `장부를 잃어버렸다고 가정했을 때 다시 펼쳐 놓을 수 있는지 연습`하는 과정입니다.

## 2. 원칙

1. 운영 Firestore를 바로 덮지 않는다.
2. 먼저 현재 상태를 별도 백업한다.
3. 가능하면 테스트 환경 또는 안전한 복구 대상에서 먼저 연습한다.
4. 복구 후에는 꼭 로그인/로드/저장까지 smoke 검증한다.
5. 리허설 결과는 반드시 문서와 리포트에 남긴다.

## 3. 리허설 대상

  - 복구 후 로그 기록이 다시 정상 생성되는지에 더 초점을 둔다.

## 5. 리허설 방식

추천은 두 단계입니다.

### 1단계. 비교 리허설
운영 Firestore를 건드리지 않고,
엑셀과 현재 Firestore 차이를 확인하는 단계입니다.

실행:
```powershell
cd D:\MYDOC\Desktop\vscode\Web\firebase\ncore_web\ncore-main
node .\scripts\compare-firestore.js
```

목적:
- 엑셀과 Firestore 차이를 파악
- 운영 중 추가된 정상 데이터와 비정상 차이를 구분

판정:
- 차이 원인이 설명 가능하면 통과
- 설명 불가 mismatch가 많으면 보류

### 2단계. 실제 복구 리허설
안전한 대상 환경에서 import를 실제 실행하는 단계입니다.

권장 대상:
1. 별도 테스트 프로젝트
2. 또는 별도 테스트 컬렉션/네임스페이스

운영 Firestore 직접 복구는 마지막에만 고려합니다.

실행:
```powershell
cd D:\MYDOC\Desktop\vscode\Web\firebase\ncore_web\ncore-main
node .\scripts\import-firestore.js
```

## 6. 상세 절차

### Step 1. 현재 운영 상태 백업

리허설 전 현재 Firestore 상태를 별도 보관한다.

예시:
- 컬렉션 export
- JSON dump
- 또는 최소한 비교 리포트 보관

남겨야 할 것:
- 실행 시각
- 대상 컬렉션 수량
- 주요 샘플 문서 ID

### Step 2. 비교 실행

```powershell
node .\scripts\compare-firestore.js
```

확인할 것:
- `missingInFirestore`
- `extraInFirestore`
- `mismatched`

메모 기준:
- 운영 중 추가된 문서는 정상으로 본다
- 예: users / requests / boardPosts / accessLogs의 Firestore 추가분

### Step 3. 테스트 복구 환경 준비

준비할 것:
- 별도 테스트 Firestore 대상
- 또는 현재 운영과 분리된 리허설 공간

주의:
- 운영 Firestore에 바로 import하지 않는다

### Step 4. import 실행

```powershell
node .\scripts\import-firestore.js
```

확인할 것:
- 컬렉션별 쓰기 성공 로그
- 오류 없이 종료되는지

### Step 5. 복구 후 smoke 검증

최소 검증:
1. 로그인
2. `session_boot`
3. `load_data`
4. 연차 요청 생성
5. 연차 요청 삭제
6. 게시글 생성/삭제

가능하면:
- [playwright-ui-smoke.js](/D:/MYDOC/Desktop/vscode/Web/firebase/ncore_web/ncore-main/scripts/playwright-ui-smoke.js)
- 또는 기존 E2E 일부 재실행

### Step 6. 복구 후 compare 재실행

```powershell
node .\scripts\compare-firestore.js
```

판정 기준:
- 설명 가능한 차이만 남으면 통과
- 예상치 못한 mismatch가 다수면 실패

## 7. 성공 기준

아래를 만족하면 리허설 통과로 본다.

1. import 스크립트가 오류 없이 종료
2. 핵심 컬렉션이 생성/갱신됨
3. 로그인/로드/신청 저장이 정상
4. compare 결과에서 치명적 mismatch 없음
5. 결과 문서와 리포트 작성 완료

## 8. 실패 시 대응

실패 유형별 대응:

### 1. import 실패
- 서비스 계정 경로
- 엑셀 경로
- 필드 normalize 로직 확인

### 2. 로그인/로드 실패
- `functions/index.js`
- 세션 토큰 발급
- `load_data` 응답 확인

### 3. 데이터 mismatch 다수
- 엑셀 normalize 기준 재확인
- Firestore 문서에 운영 중 추가된 필드/문서 분리 판단
