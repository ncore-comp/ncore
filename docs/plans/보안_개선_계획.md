# 보안 개선 계획

## 1. 목적

- NCORE 연차관리 시스템의 보안 취약점을 운영 가능한 수준에서 순차적으로 줄인다.
- `즉시 고쳐야 하는 것`과 `조금 더 나눠서 진행해도 되는 것`을 구분한다.
- 기능 화면을 깨지 않으면서, 권한과 데이터 응답 범위를 더 정확하게 맞춘다.

쉽게 말하면:
- 지금까지는 건물에 잠금장치를 하나씩 달아온 단계였고,
- 이 문서는 어떤 문을 먼저 잠그고 어떤 서류를 어디까지 나눠 줄지 정리한 공사 일정표다.

## 2. 초기에 확인된 핵심 위험

1. 비밀번호가 평문으로 저장/비교되고 있었음
2. 로그인한 사용자에게 필요 이상으로 넓은 데이터가 내려가고 있었음
3. 승인권한과 운영실 관리자 수정권한이 일부 섞여 있었음
4. 로그인 시도 제한이 없어 반복 대입 공격에 취약했음
5. 현재 비밀번호 확인 없이 비밀번호를 변경할 수 있었음

## 3. 우선순위

### 즉시 고쳐야 하는 것

- 비밀번호 해시 전환
- 비밀번호 변경 재인증
- 로그인 시도 제한
- 승인권한과 운영실 수정권한 분리

### 나눠서 진행해도 되는 것

- `load_data` 응답 최소화
- 운영실/메일/특별휴가/게시판 데이터 분리
- 화면별 전용 데이터 로드 구조로 개편
- 세션 정책 추가 강화
- 보안 로그 체계화

## 8. 현재 남은 보안 이슈 정리 (2026-03-27 11:24)

현재 기준 판단:
- 치명급 큰 구멍은 대부분 막힌 상태
- 남은 것은 `보안 하드닝` 단계

쉽게 말하면:
- 문이 열려 있는 단계는 지났고,
- 지금은 `이중 잠금`, `출입카드 도난 대비`, `경비 사고장부 분리`가 남은 상태다.

### 남은 핵심 이슈

1. 세션 강화
- 세션 생성 시 `createdIp`는 저장하지만, 실제 요청에서 IP/UA 이상징후를 적극 비교하지는 않음
- 세션 토큰 유출 시 다른 환경 재사용 대응 여지가 남아 있음

2. 보안 로그 체계화
- 현재는 대부분 `accessLogs` 중심
- 로그인 실패/차단/권한 변경/운영실 주요 작업을 `securityLogs`처럼 따로 보진 않음

3. 비밀번호 fallback 제거
- 현재 운영 데이터는 해시 전환 완료 상태지만
- `passwordHash/passwordSalt`가 없을 경우 평문 비교 fallback 코드가 남아 있음

4. 로그인 차단 정책 2차 강화 여부
- 현재는 `같은 아이디 + 같은 IP` 기준 5회 / 5분 차단
- 더 강한 분산형 방어는 아직 미적용

5. load_data 3차 잔여 작업
- 게시판 / 전체 상황판은 전용 로드 분리 완료
- 팀원 연차현황 / 팀 달력 / 내 대시보드 쪽은 추가 분리 가능

### 한 줄 정리

- 지금 남은 보안 이슈의 중심은 `세션 강화`와 `보안 로그 체계화`다.
