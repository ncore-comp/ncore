# 잔업/특근 반려 후 제출 대기 잔존 개선 계획
# Overtime/Holiday Work Rejection Still Showing as Pending Plan

## 문제 요약
## Problem Summary

- 특근 보고를 반려한 뒤에도 홈화면의 `잔업/특근 제출 대기` 카드에 계속 남는다.
- After a work report settlement is rejected, it still remains in the home dashboard `Pending Overtime/Holiday Work Submission` card.

- 다시 반려를 누르면 서버는 이미 `pending`이 아니라고 응답한다.
- If the user tries to reject it again, the server responds that it is no longer `pending`.

- 즉 서버 상태와 프런트 표시 기준이 어긋난 상태다.
- In other words, the server state and the frontend display logic are out of sync.

## 확인된 원인
## Confirmed Root Cause

1. 서버 `requestService`는 반려 시 `settlementStatus: "rejected"`를 저장한다.
1. The server `requestService` stores `settlementStatus: "rejected"` on rejection.

2. 그러나 `loadService`가 요청 목록을 내려줄 때 `settlementStatus`를 포함하지 않는다.
2. However, `loadService` does not include `settlementStatus` when returning request data.

3. 프런트 `ui_workreport_v127_codex.js`는 `settlementStatus`가 비어 있으면 `settlementSubmittedAt`만 보고 `submitted`로 간주한다.
3. In `ui_workreport_v127_codex.js`, when `settlementStatus` is empty, the frontend treats any request with `settlementSubmittedAt` as `submitted`.

4. 그래서 반려된 건도 홈에서는 계속 `제출 대기`로 보이게 된다.
4. As a result, rejected settlements still appear as `pending submission` on the home dashboard.

## 개선 목표
## Improvement Goal

- 반려/승인/제출 상태가 서버 저장값 기준으로 일관되게 보이게 한다.
- Make submitted/approved/rejected status display consistently based on the server-stored state.

- 홈화면과 상세화면이 같은 정산 상태 판정 기준을 사용하게 한다.
- Ensure the home dashboard and detail screens use the same settlement status interpretation.

- 반려된 건은 `잔업/특근 제출 대기`에서 즉시 사라지게 한다.
- Make rejected settlements disappear from `Pending Overtime/Holiday Work Submission` immediately.
