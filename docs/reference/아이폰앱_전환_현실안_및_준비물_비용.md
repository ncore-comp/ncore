# 아이폰 앱 전환 현실안 및 준비물/기간/비용

## 1. 결론 먼저

현재 프로젝트 기준으로 **가장 현실적인 아이폰 앱 전환 방식은 `웹뷰 앱(Capacitor 방식)`**이다.

쉽게 비유하면:
- 지금 잘 돌아가는 웹 매장을 그대로 두고,
- `아이폰 전용 출입문`과 `앱스토어 간판`만 새로 다는 방식이다.

즉:
- Firebase 백엔드 유지
- 현재 웹 UI 최대한 재사용
- iPhone 앱 껍데기만 추가

이 방식이 가장 빠르고, 비용/리스크가 가장 낮다.

---

## 2. 선택지 비교

### 1안. PWA

설명:
- 지금 웹을 `홈 화면에 추가`해서 앱처럼 쓰는 방식

장점:
- 가장 빠름
- 개발량 적음
- 비용 적음

단점:
- App Store 등록과는 별개
- 사용자가 앱스토어에서 설치하는 일반 앱 느낌은 약함
- iPhone에서 제약이 있음

판단:
- `앱처럼 쓰기`는 가능하지만
- `앱스토어 등록`을 목표로 하면 메인 답은 아님

---

### 2안. 웹뷰 앱 (Capacitor)  
추천

설명:
- 현재 웹을 iOS 앱 껍데기 안에 넣어서 앱처럼 배포

장점:
- 현재 코드 재사용이 가장 많음
- Firebase 구조 그대로 가져가기 좋음
- App Store 배포 가능
- Android 확장도 쉬움

단점:
- Mac / Xcode 필요
- 모바일 UX 미세조정은 추가로 필요
- 웹에서 잘 되던 일부 기능은 앱 기준으로 다시 손봐야 할 수 있음

판단:
- 지금 프로젝트에 가장 현실적
- 비용/시간/리스크 균형이 가장 좋음

---

### 3안. 네이티브 재개발

설명:
- Swift / React Native / Flutter 등으로 새 앱을 다시 만드는 방식

장점:
- 가장 완성도 높게 갈 수 있음
- 모바일 UX 최적화에 유리

단점:
- 개발량 큼
- 기간 길어짐
- 비용 크게 증가
- 지금 만든 웹 자산 재사용률이 낮아짐

판단:
- 현재 단계에서는 과함

---

## 3. 현재 프로젝트에 2안이 맞는 이유

현재 시스템은:
- Firebase Hosting + Functions + Firestore
- 브라우저 기반 UI
- 모바일 보조 UI 코드 일부 존재

즉 이미 `모바일에서 읽히는 웹`의 뼈대는 있다.

비유하면:
- 바닥 공사와 전기 배선은 이미 끝난 상태라
- 새로 건물을 짓기보다 `앱 외벽`만 입히는 게 맞다.

현재 기준 장점:
1. 로그인/권한/연차/운영실 흐름 재사용 가능
2. 서버 로직을 다시 만들 필요가 적음
3. iPhone과 Android를 같은 방향으로 묶어가기 쉬움

---

## 4. 실제 진행 방식

아이폰 앱 전환은 보통 이 순서로 간다.

1. 현재 웹을 앱셸에 넣는다
2. Capacitor로 iOS 프로젝트 생성
3. Mac + Xcode에서 빌드
4. iPhone 실기기 테스트
5. TestFlight 배포
6. App Store 제출

즉:
- 웹을 먼저 버리지 않는다
- 웹을 기준으로 앱 껍데기를 씌운다

---

## 5. 준비물

### 필수

1. Mac
   - iOS 빌드는 사실상 Mac 필요

2. Xcode
   - iOS 앱 빌드 도구

3. Apple Developer Program 계정
   - 앱스토어 등록용 유료 계정

4. 현재 프로젝트 소스
   - 이 웹 프로젝트 그대로 사용

5. 앱 정보
   - 앱 이름
   - 아이콘
   - 스플래시 이미지
   - 번들 ID
   - 설명문
   - 스크린샷

### 있으면 좋은 것

1. 테스트용 iPhone 실기기
2. 조직 계정 정보
3. 개인정보처리방침 URL

## 6. 예상 추가 개발 항목

웹을 그대로 앱으로 감쌀 수는 있지만, 아래는 따로 손볼 가능성이 높다.

1. 로그인 유지 방식
2. 메일 호출 방식
3. CSV 다운로드 방식
4. 파일 열기/공유 방식
5. 모달 크기와 버튼 터치 영역
6. 모바일 달력 조작감
7. 상단 네비/탭 크기

즉:
- `동작은 된다`
- 하지만 `앱답게 다듬는 작업`은 필요하다

범위:
- 앱 정보 작성
- 스크린샷

예상:
- `3일 ~ 7일 + 심사기간`

## 8. 예상 비용

### 내부 진행 시

1. Apple Developer Program
   - 연간 비용 발생

2. 개발 공수
   - 웹뷰 앱화 + 모바일 보정

체감상:
- 최소 `수십만 원 + 개발 공수`

### 외주 관점 체감

1. 아주 최소 앱 포장만
   - 비교적 저렴

2. 실사용 앱 수준 보정까지 포함
   - 더 커짐

현실적으로는:
- `간단 앱 포장`: 수백만 원대 가능
- `실사용 보정 포함`: 그 이상

즉 웹을 이미 만들어둔 덕분에,
처음부터 앱을 새로 만드는 것보다는 훨씬 싸다.

## 9. 리스크

1. Mac이 없으면 iOS 빌드가 막힘
2. 앱스토어 심사 기준 대응 필요
3. 웹에선 자연스러운 기능이 앱에서 다르게 보일 수 있음
   - 다운로드
   - 메일
   - 세션 처리

즉:
- 가장 큰 기술 리스크는 `앱 포장`보다 `모바일 UX 보정`이다

## 11. 한 줄 결론

현재 프로젝트를 아이폰 앱으로 만드는 가장 현실적인 방법은  
**`기존 웹을 Capacitor 같은 웹뷰 앱 방식으로 감싸고, Mac/Xcode에서 iOS 빌드를 만들어 TestFlight와 App Store로 올리는 것`**이다.

즉:
- 새 집을 짓는 게 아니라
- 지금 집에 `아이폰용 현관문`을 다는 방식이 가장 맞다.
